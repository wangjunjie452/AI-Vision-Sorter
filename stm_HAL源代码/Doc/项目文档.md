## 1. 项目概述与核心功能

### 1.1 项目定位

本项目是 **STM32F103C8T6 的 HAL 库空白模板工程**，旨在提供一个开箱即用的 STM32 开发起点。项目已预配置好以下基础设施：

- 系统时钟（HSE 8MHz → PLL ×9 = 72MHz）
- 调试串口（USART1，115200 8N1，printf 重定向）
- 精确延时（DWT 硬件计数器，微秒/毫秒/秒级）
- RGB LED 驱动（PA1/PA2/PA3，共阳）
- 中断服务框架（SysTick 时基 + 异常处理）

### 1.2 核心功能清单

| 功能模块 | 描述 | 文件 |
|---|---|---|
| 时钟配置 | 外部 8MHz 晶振 → PLL 倍频至 72MHz；AHB=72M, APB1=36M, APB2=72M | `User/main.c` |
| 调试串口 | USART1 PA9/PA10，115200-8N1；`printf()` 重定向至串口 | `User/usart/bsp_usart.c` |
| 精确延时 | 基于 Cortex-M3 DWT 寄存器（`DWT_CYCCNT`），理论精度 14ns | `User/dwt/bsp_dwt.c` |
| RGB LED | 三色共阳 LED，低电平点亮，宏控制开关/翻转/组合 | `User/led/bsp_led.c` |
| 中断框架 | SysTick 维持 HAL 时基，异常 Handler 均已声明 | `User/stm32f1xx_it.c` |

### 1.3 引脚分配

| 功能 | 信号 | 引脚 | 模式 |
|---|---|---|---|
| 调试串口 TX | USART1_TXD | PA9 | 复用推挽输出 |
| 调试串口 RX | USART1_RXD | PA10 | 浮空输入 |
| RGB 红灯 | R_LED | PA1 | 推挽输出（低电平亮） |

## 4. 各模块的功能与职责解析

### 4.1 系统启动模块

**涉及文件**：`Libraries/CMSIS/Device/ST/STM32F1xx/Source/Templates/arm/startup_stm32f103xb.s`、`User/system_stm32f1xx.c`

```
上电/复位
  │
  ├─→ 初始化 MSP（从向量表加载主堆栈指针）
  ├─→ 调用 SystemInit()
  │      ├─ 使能 HSI 时钟（如果 HSE 不可用则使用 HSI）
  │      ├─ 配置 Flash 等待周期
  │      └─ 配置 AHB/APB 预分频器为默认值（HSI/HSE/8）
  └─→ 跳转 main()（由 __main 调用 main）
```

`SystemCoreClock` 变量初始值为 **8,000,000**（8MHz），后续由 `SystemClock_Config()` 内部的 `HAL_RCC_ClockConfig()` 自动更新为 72MHz。

### 4.2 时钟配置模块

**涉及文件**：`User/main.c` 中的 `SystemClock_Config()`

| 参数 | 值 |
|---|---|
| 时钟源 | HSE（外部 8MHz 晶振） |
| PLL 源 | HSE 不分频（`RCC_HSE_PREDIV_DIV1`） |
| PLL 倍频 | ×9（`RCC_PLL_MUL9`） |
| SYSCLK | 72MHz |
| AHB 预分频 | ÷1（HCLK = 72MHz） |
| APB1 预分频 | ÷2（PCLK1 = 36MHz） |
| APB2 预分频 | ÷1（PCLK2 = 72MHz） |
| Flash 等待 | 2 WS（72MHz 要求） |

执行流程：

```
SystemClock_Config()
  ├─ HAL_RCC_OscConfig()       → 启用 HSE + PLL，等待就绪
  └─ HAL_RCC_ClockConfig()     → 切换到 PLL 时钟源
       ├─ 设置 AHB/APB1/APB2 预分频
       ├─ 配置 Flash 等待周期
       ├─ 选择 PLL 作为系统时钟
       ├─ 等待 PLL 就绪
       └─ 调用 SystemCoreClockUpdate() → 将 SystemCoreClock 更新为 72MHz
```

### 4.3 DWT 精确延时模块

**涉及文件**：`User/dwt/bsp_dwt.h`、`User/dwt/bsp_dwt.c`

#### 工作原理

DWT（Data Watchpoint and Trace）是 Cortex-M 内核内置的调试外设，包含一个 32 位向上计数器 `DWT_CYCCNT`，直接记录 CPU 执行过的时钟周期数：

- 72MHz 下，每个 tick ≈ 13.9ns
- 32 位计数器可记录最长约 60 秒（2³² / 72MHz）

#### 初始化（`DWT_Init`）

```c
DEMCR |= DEMCR_TRCENA;          // 位24，使能 DWT 调试跟踪
DWT_CYCCNT = 0;                 // 计数器清零
DWT_CTRL |= DWT_CTRL_CYCCNTENA; // 位0，启动 CYCCNT 计数
```

#### 延时计算（`DWT_DelayUs`）

```c
tick_duration = time × (SystemCoreClock / 1,000,000)
// 例：1000µs × (72000000 / 1000000) = 1000 × 72 = 72000 ticks
```

然后空转等待 `DWT_CYCCNT` 增加足够差值。

#### API 一览

| 函数 | 功能 | 参数 |
|---|---|---|
| `DWT_Init()` | 初始化并启动 DWT 计数器 | 无 |
| `DWT_GetTick()` | 读取当前 CYCCNT 值 | 无 |
| `DWT_DelayUs(uint32_t us)` | 微秒延时 | 延时微秒数 |
| `DWT_DelayMs(uint32_t ms)` | 毫秒延时 | 延时毫秒数 |
| `DWT_DelayS(uint32_t s)` | 秒级延时 | 延时秒数 |

#### 仿真注意事项

Keil 仿真环境下 PLL 不被模拟，`SystemCoreClock` 保持默认 8MHz，导致延时为预期的 9 倍。解决方式：在 `SystemClock_Config()` 之后添加 `SystemCoreClock = 72000000;`，真机不受影响（HAL 内部会覆盖）。

### 4.4 RGB LED 驱动模块

**涉及文件**：`User/led/bsp_led.h`、`User/led/bsp_led.c`

#### 硬件特性

开发板使用 **共阳 RGB LED**：公共端接 VCC（高电平），三色阴极分别接 PA1/PA2/PA3。因此 **GPIO 输出低电平时点亮，高电平时熄灭**。

#### 引脚宏

```c
#define R_LED_Pin   GPIO_PIN_1   // 红灯 → PA1
#define G_LED_Pin   GPIO_PIN_2   // 绿灯 → PA2
#define B_LED_Pin   GPIO_PIN_3   // 蓝灯 → PA3
#define LED_Port    GPIOA
```

#### 控制宏（直接调用 HAL API）

```c
// 单色控制
R_LED_ON()      → HAL_GPIO_WritePin(GPIOA, PA1, RESET)   // 亮
R_LED_OFF()     → HAL_GPIO_WritePin(GPIOA, PA1, SET)     // 灭
R_LED_TOGGLE()  → HAL_GPIO_TogglePin(GPIOA, PA1)          // 翻转

// 组合控制
RGB_ALL_ON()    → 三色全亮（白光）
RGB_ALL_OFF()   → 三色全灭
R_LED_ON_ONLY() → 仅亮红灯（绿灯、蓝灯灭）
G_LED_ON_ONLY() → 仅亮绿灯
B_LED_ON_ONLY() → 仅亮蓝灯
```

#### 初始化（`LED_GPIO_Config`）

```
1. __HAL_RCC_GPIOA_CLK_ENABLE()    使能 GPIOA 时钟
2. HAL_GPIO_Init()                 配置 PA1/PA2/PA3 为推挽输出、低速、无上下拉
3. HAL_GPIO_WritePin(..., SET)     默认全灭（输出高电平）
```

### 4.5 调试串口模块

**涉及文件**：`User/usart/bsp_usart.h`、`User/usart/bsp_usart.c`

#### 配置参数

| 参数 | 值 |
|---|---|
| 外设 | USART1 |
| 波特率 | 115200 |
| 数据位 | 8 bit |
| 停止位 | 1 bit |
| 校验 | 无（None） |
| 硬件流控 | 无（None） |
| 过采样 | 16 倍 |
| 模式 | 发送 + 接收（TX/RX） |

#### MSP 模式

HAL 库使用 MSP（MCU Support Package）模式，`HAL_UART_Init()` 内部自动调用 `HAL_UART_MspInit()` 完成：

1. 使能 `USART1` 时钟 → `__HAL_RCC_USART1_CLK_ENABLE()`
2. 使能 `GPIOA` 时钟 → `__HAL_RCC_GPIOA_CLK_ENABLE()`
3. 配置 PA9（TX）→ 复用推挽输出，高速
4. 配置 PA10（RX）→ 浮空输入

同理，`HAL_UART_DeInit()` 会自动调用 `HAL_UART_MspDeInit()` 释放 GPIO 并关闭时钟。

#### printf 重定向

```c
int fputc(int ch, FILE *f)
{
    HAL_UART_Transmit(&huart1, (uint8_t *)&ch, 1, HAL_MAX_DELAY);
    return ch;
}
```

标准库的 `printf()` 最终调用 `fputc()`，将字符逐一通过 USART1 阻塞发送。注意使用的是 **阻塞模式**（`HAL_UART_Transmit`），发送完成前不返回。

### 4.6 中断服务模块

**涉及文件**：`User/stm32f1xx_it.h`、`User/stm32f1xx_it.c`

#### 已实现的异常 Handler

| 异常 | 优先级 | 行为 |
|---|---|---|
| `NMI_Handler` | -2（固定） | 空函数（占位） |
| `HardFault_Handler` | -1（固定） | 死循环（方便调试时定位） |
| `MemManage_Handler` | 可配置 | 死循环 |
| `BusFault_Handler` | 可配置 | 死循环 |
| `UsageFault_Handler` | 可配置 | 死循环 |
| `SVC_Handler` | 可配置 | 空函数 |
| `DebugMon_Handler` | 可配置 | 空函数 |
| `PendSV_Handler` | 可配置 | 空函数 |
| `SysTick_Handler` | 可配置 | 调用 `HAL_IncTick()` 维持 HAL 时基 |

#### SysTick 时基

`SysTick_Handler` 是**唯一有业务逻辑的中断**，每 1ms 触发一次（HAL 默认配置），调用 `HAL_IncTick()` 递增全局 tick 计数器，供 `HAL_Delay()` 使用。

**注意**：本项目使用 DWT 延时（`DWT_DelayMs`），而非 `HAL_Delay()`。DWT 精度更高且不依赖 SysTick。

## 5. 关键代码逻辑说明

### 5.1 main() 完整执行流程

```c
int main(void)
{
    HAL_Init();             // ① 初始化 HAL 库（NVIC 优先级分组、SysTick 配置）
    SystemClock_Config();   // ② 配置系统时钟 72MHz（HSE + PLL）
    Debug_UART_Init();      // ③ 初始化调试串口 USART1（115200-8N1）
    LED_GPIO_Config();      // ④ 初始化 RGB LED 引脚（PA1/PA2/PA3）
    DWT_Init();             // ⑤ 启动 DWT 精确延时计数器

    while (1)               // ⑥ 主循环
    {
        // 应用逻辑在此实现
    }
}
```

**调用时序**：串口、LED 的初始化必须在 DWT 之前，以确保 DWT 计时不受外设初始化过程的影响（但实际影响可忽略）。

### 5.2 SystemClock_Config() 错误处理逻辑

```c
// 步骤1：配置振荡器（HSE + PLL）
if (HAL_RCC_OscConfig(&oscinitstruct) != HAL_OK)
{
    while(1);   // HSE 晶振故障则死循环（硬件级错误）
}

// 步骤2：切换系统时钟源到 PLL
if (HAL_RCC_ClockConfig(&clkinitstruct, FLASH_LATENCY_2) != HAL_OK)
{
    while(1);   // PLL 锁定失败则死循环（极少发生）
}
```

两阶段检查：
1. **振荡器配置失败**：通常意味着外部晶振没焊接或损坏
2. **PLL 切换失败**：Flash 等待周期配置错误或 PLL 参数非法

**禁止**在未处理错误时继续执行后续代码（符合 HAL 规范第 10 条）。

### 5.3 DWT_DelayUs() 算法详解

```c
void DWT_DelayUs(uint32_t time)
{
    uint32_t tick_duration = time * (SystemCoreClock / 1000000);
    uint32_t tick_start    = DWT_GetTick();

    while (DWT_GetTick() - tick_start < tick_duration);
}
```

| 步骤 | 操作 | 示例（72MHz，延时 500µs） |
|---|---|---|
| 1 | 计算所需 tick 数 | 500 × 72 = 36000 ticks |
| 2 | 记录起始 tick 值 | `tick_start = 12345` |
| 3 | 空转等待 | 循环直到 CYCCNT ≥ 48345 |

**理论最小延时**：1µs（72 ticks ≈ 1µs @ 72MHz）  
**精度误差**：±1 个 tick（约 14ns），主要来源于 `DWT_GetTick()` 调用和减法比较的开销  
**溢出处理**：由于使用无符号 32 位减法（`now - start < duration`），计数器溢出自动正确处理（利用无符号整数环绕特性）

### 5.4 DWT_DelayMs() 实现方式

```c
void DWT_DelayMs(uint32_t time)
{
    for (uint32_t i = 0; i < time; i++)
    {
        DWT_DelayUs(1000);   // 每次延时 1000µs = 1ms
    }
}
```

**不是**直接计算 `time * 72000` ticks，而是通过循环 1000µs × time 次实现。原因：
- 避免 `tick_duration` 溢出（32 位变量在 72MHz 下最大只能表示约 59ms）
- 循环开销可忽略（每次循环仅几条指令）

### 5.5 HAL_UART_MspInit() 回调机制

```
Debug_UART_Init()
  └─ HAL_UART_Init(&huart1)
       ├─ 参数校验（assert_param）
       ├─ 设置 UART 状态为 BUSY
       ├─ HAL_UART_MspInit(huart1)   ← 用户实现，配置 GPIO 和外设时钟
       │    ├─ __HAL_RCC_USART1_CLK_ENABLE()
       │    ├─ __HAL_RCC_GPIOA_CLK_ENABLE()
       │    ├─ HAL_GPIO_Init(GPIOA, &tx_config)   // PA9
       │    └─ HAL_GPIO_Init(GPIOA, &rx_config)   // PA10
       ├─ 配置 UART 寄存器（波特率、数据位等）
       └─ 清除 UART 状态标志
```

MSP 模式使得硬件相关初始化（GPIO/时钟）与外设逻辑配置（波特率/数据位）分离，便于在不同硬件上移植。

### 5.6 错误处理（Error_Handler）

```c
void Error_Handler(void)
{
    __disable_irq();   // 全局关中断
    while (1) { }      // 死循环，等待调试器介入
}
```

- 关中断确保不会因中断扰乱调试现场
- 死循环方便在调试器中暂停并查看调用栈
- 符合 HAL 规范第 4 条（关键函数必须检查返回值）

## 6. 项目的启动与部署步骤

### 6.1 环境准备

1. **安装 Keil MDK-ARM V5**（建议 V5.25 以上）
2. **安装器件包**：在 Keil 的 Pack Installer 中搜索并安装 `Keil.STM32F1xx_DFP`（≥2.2.0）
3. **安装 ST-Link 驱动**（如果用 ST-Link 调试器）
4. **硬件连接**：
   - USB 转串口连接 PA9/PA10（或直接使用开发板 USB 口）
   - ST-Link/J-Link 连接 SWDIO/SWCLK/GND

### 6.2 打开工程

双击 `Project/Fire_F103.uvprojx` 在 Keil µVision5 中打开工程。

### 6.3 编译

- **快捷键**：`F7`（Build）
- **重新编译**：`Ctrl + Alt + F7`（Rebuild all）
- **编译产物**：
  - `Project/Objects/Fire_F103.axf` — 带调试信息的 ELF 文件
  - `Project/Objects/Fire_F103.hex` — 烧录用 Hex 文件

预期编译结果：0 错误，0 警告。

### 6.4 仿真调试（可选）

1. 在 Keil 工具栏中选择调试器为 "Use Simulator"
2. 点击 `Ctrl + F5` 进入 Debug 模式
3. **重要**：仿真前在 `main()` 的 `SystemClock_Config()` 后添加：
   ```c
   SystemCoreClock = 72000000;  // Keil 仿真器不模拟 PLL，需手动设置
   ```
4. 可使用逻辑分析仪（Logic Analyzer）观察 GPIO 引脚波形

### 6.5 烧录到芯片

1. 连接 ST-Link 或 J-Link 调试器
2. Keil 中配置 "Debug → Use: ST-Link Debugger"（或 J-Link）
3. 在 "Utilities → Use Debug Driver" 勾选并选择对应的 Flash 算法
4. 点击 `F8`（Download）或 `Ctrl + F5`（烧录并进入调试）

### 6.6 验证运行

1. **串口验证**：打开串口调试助手（如 SSCOM、PuTTY）
   - 波特率：115200
   - 数据位：8，停止位：1，校验：None
   - 端口：对应 USB 转串口 COM 号
2. **LED 验证**：在 `main()` 的 while 循环中添加 LED 控制代码，观察 LED 闪烁。

### 6.7 添加新模块

在 `APP` 目录（或 `User` 下新建子目录）中创建 `.c/.h` 文件，然后：

1. 在 Keil 工程 "APP" Group 中添加 `.c` 文件
2. 在使用的源文件中 `#include` 对应头文件
3. 如需向 HAL 添加新外设支持，修改 `stm32f1xx_hal_conf.h` 取消对应模块注释
4. 在 `Libraries/STM32F1xx_HAL_Driver/Src/` 中找到对应 `.c` 文件加入工程
